## Phase 4: Cover System Rebuild (Apr 19 ~14:30–15:55)

The cover system had fundamental problems — enemies couldn't find cover points at runtime because group membership wasn't persisting:

- **cover_point group persistence** — `add_to_group('cover_point')` loses group membership on scene save/load. Fix: `add_to_group('cover_point', true)` with `persistent=true`.
- **CoverBuilder integrated into walk_scene** — Added CoverBuilder node to the scene, placed cover props manually.
- **Cover facing system** — CoverBuilder now orients cover point Node3D so -Z faces the obstacle. Stores `facing` meta (toward obstacle) and `cover_type` ('full'/'half'). AI cover scoring uses facing dot product: facing toward player = +12, facing away = -25.
- **Stop Standard/Heavy from rushing** — When no cover is found, Standard/Heavy now expand search radius by 5m every 2s (up to 40m) instead of just advancing to the player's face. Only Rusher advances after 3s timeout.
- **Debug improvements** — Cover spheres colored by type: blue=full, green=half, red=claimed. Editor debug arrows show facing direction.

## Phase 5: AI V3 Design & Implementation (Apr 19 ~16:00)

The biggest change of the day — AI V3 fixes the core problem of enemies not using their weapons intelligently:

- **AI V3 Fix Plan** (AI_V3_FIX_PLAN.md) — Documented weapon engage distance, Rusher redesign, peek animation issues, and fireteam debug needs. 275 lines of design.
- **Weapon engage distance** — Each archetype has an `ideal_engage_distance`: Rusher 10m, Standard 15m, Heavy 25m. Cover scoring rewards positions at the right range (+15 sweet spot, +5 acceptable, -10 wrong range). ADVANCE state: all archetypes seek cover when within engage_distance + 5m.
- **Rusher redesign** — No longer charges blindly to melee range. Uses cover like everyone else, just closer cover (10m ideal). Pushes forward cover-to-cover when teammates suppress. Peek-shoots from cover with shotgun. Only melees reactively within 2m.
- **Fireteam boost** — Clustering bonus +8 -> +15 (within 3-8m of teammate). Disconnection penalty -5 -> -8 (>15m). FT0 prefers cover in front of player (+8), FT1 prefers flanking positions (+8).
- **Tactical minimap** (F3 toggle) — 300px top-down map in top-right corner. Blue dots = FT0, Orange dots = FT1. Dot size by archetype. Lines to cover targets. Yellow ring = holding engagement slot. Phase indicator text. 10m grid. Player as white triangle with facing direction.

## Phase 6: Cover Peek Rewrite (Apr 20 morning)

Enemies were shaking violently during peek-shoot because `velocity` and `look_at_player` fought each other every frame:

- **3-phase peek system** — PEEK_OUT (move to peek position, no aiming), PEEK_SHOOT (stationary, smooth rotation toward player, then fire), PEEK_RETURN (move back to cover, no aiming). IN_COVER now sets velocity=0 and does NOT look at player.
- **Cover debug heights** — Half cover spheres at 0.7m (crouch eye height), full cover at 1.4m (standing eye height).
- **F5 free camera fix** — Now fully disables player input (physics + input + unhandled_input + process). Mouse look works properly in free cam.
- **AI_COVER_PEEK_ANALYSIS.md** — Documented how Division, Gears, and FEAR handle cover peek for future reference.

## Architecture Decisions

| Decision | Rationale |
|---|---|
| Weapon engage distance per archetype | Enemies need to fight at their weapon's effective range, not all cluster at the same distance |
| Rusher uses cover (just closer) | Blind charging was a guaranteed death sentence; cover-to-cover advance is tactically credible |
| 3-phase peek (out/shoot/return) | Single-state peek caused velocity vs look_at fighting; separating movement from aiming eliminates jitter |
| Cover group persistent=true | Without it, cover points only existed at edit time; runtime code saw zero cover |
| Per-weapon VFX identity | Players need to visually distinguish weapons at a glance; color/size/trail make each feel unique |
| Tactical minimap | Impossible to understand squad behavior from first-person view; top-down map essential for AI debugging |
